No real security issue.

### first-repo — CLEAN (all 4 CRITICAL findings are false positives)
The scanner flagged 4 CRITICAL issues (hardcoded `"test_token"` strings) across:
- `scripts/test_integration.py:95` — test script only
- `scripts/verify_system_observer.py:56,238` — verify/diagnostic script only
- `scripts/run_system_observer.py:171` — only under `--no-telegram` flag (explicit test mode)

All are dummy placeholders in test/diagnostic code paths, never used in production.
No real credentials hardcoded. **No action required.**

The 101 MEDIUM hits across first-repo are all false positives from the `print_credentials` pattern
matching on benign uses of the words "key", "keys", "token" in print statements for debug output
and error messages. Pattern is too broad for first-repo's extensive diagnostic/test scripts.

**Recommendation:** Consider tightening the `print_credentials` pattern to require a value being
printed alongside the credential word, e.g. `print.*(?:token|key)\s*=\s*\S+` rather than just
the presence of the word. This would eliminate the false positive flood in first-repo.
